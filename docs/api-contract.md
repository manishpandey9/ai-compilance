# API Contract — EU AI Act Compliance Navigator

| | |
|---|---|
| **Companion to** | [`technical-design.md`](./technical-design.md) §9 |
| **Source PRD** | §14.5 (endpoints), §10 (functional reqs) |
| **Base URL** | `https://api.<domain>/api/v1` |
| **Format** | JSON over HTTPS · OpenAPI 3.1 auto-generated by FastAPI |
| **Status** | Draft for review |

The FastAPI `schemas.py` (Pydantic v2) is the **single source of truth**; the Next.js typed client is generated from the emitted OpenAPI spec (TDD §5.2). This document is the human-readable contract; CI asserts client/spec parity (TDD §15).

---

## 1. Conventions

### 1.1 Versioning & headers
- All routes are under `/api/v1`. Breaking changes bump the prefix.
- `Authorization: Bearer <jwt>` for authenticated routes (Clerk-issued, verified via JWKS).
- `Idempotency-Key: <uuid>` **required** on all mutating POSTs and honored on payment webhooks.
- `X-Request-Id` echoed for tracing (web ↔ api correlation).

### 1.2 Error envelope
All non-2xx responses share one shape:

```json
{ "error": { "code": "validation_error", "message": "Human readable", "details": [ { "field": "email", "issue": "required" } ] } }
```

| HTTP | `code` examples |
|---|---|
| 400 | `validation_error`, `malformed_request` |
| 401 | `unauthenticated` |
| 403 | `forbidden`, `entitlement_required` |
| 404 | `not_found` |
| 409 | `conflict`, `idempotency_replay` |
| 422 | `unprocessable` |
| 429 | `rate_limited` |
| 500 | `internal_error` |

### 1.3 Important: product states are 200, not errors
The classify endpoint returns **HTTP 200** with a typed `classification_status` for `insufficient_information`, `needs_expert_review`, and `conflicting_rules` (TDD §7.3, PRD §10.2). These are valid outcomes, never HTTP failures.

### 1.4 Pagination
List endpoints use cursor pagination: `?limit=&cursor=` → `{ "data": [...], "next_cursor": "..." }`.

### 1.5 Rate limiting
Public assessment + checkout endpoints are throttled per IP + per email; `429` with `Retry-After`.

## 2. Public — Pages (PRD §14.5, FR-SEO-*)

### `GET /pages/{slug}`
Returns a composed pSEO page record for rendering (used by ISR/build and previews).

**200**
```json
{
  "slug": "hr-tech/resume-screening",
  "page_type": "use_case",
  "title": "EU AI Act Compliance for Resume Screening AI",
  "meta_description": "Is resume screening AI high-risk under the EU AI Act? ...",
  "content_md": "## Quick answer ...",
  "structured_data": [ { "@type": "FAQPage", "...": "..." } ],
  "canonical_url": "https://.../eu-ai-act/hr-tech/resume-screening/",
  "last_reviewed_at": "2026-06-01T00:00:00Z",
  "rule_version": 7,
  "source_version": "2024/1689-consolidated-2026-01",
  "references": [ { "canonical_citation": "Regulation (EU) 2024/1689 Annex III point 4(a)", "url": "..." } ],
  "internal_links": { "industry": "/eu-ai-act/hr-tech/", "related_use_cases": ["..."], "templates": ["..."], "articles": ["..."] }
}
```
**404** if slug not `active`.

### `GET /pages?type=&industry=&risk_tier=&limit=&cursor=`
Filtered listing for hubs/sitemap. Returns paginated minimal page records (`slug`, `title`, `page_type`, `last_reviewed_at`).

## 3. Public — Assessments (PRD §14.5, §10.1)

### `POST /assessments`
Create a draft assessment (anonymous allowed).

**Request**
```json
{ "company_name": "Acme AI", "system_name": "HireRank" }
```
**201**
```json
{ "assessment_id": "aia_123", "status": "draft", "question_set_version": 3, "claim_token": "tok_..." }
```
`claim_token` lets an anonymous session later attach the assessment to a user on email capture/sign-in (TDD §7.2).

### `GET /assessments/{id}`
Returns assessment + answers + next questions (respects branching).

**200**
```json
{
  "assessment_id": "aia_123",
  "status": "in_progress",
  "answers": { "use_case_category": "employment_recruitment", "affects_natural_persons": true },
  "next_questions": [
    { "question_key": "system_function", "type": "multi", "label": "What does the system do?",
      "options": ["filter_applications","rank_candidates","evaluate_candidates"],
      "help": "Select all that apply", "allow_unknown": true }
  ],
  "progress": { "answered": 8, "remaining_estimate": 4 }
}
```

### `PATCH /assessments/{id}`
Update metadata (company/system name), claim with `claim_token`, or capture email.

**Request**
```json
{ "email": "founder@acme.ai", "marketing_consent": true, "claim_token": "tok_..." }
```

### `POST /assessments/{id}/answers`
Upsert one or more answers; returns recomputed `next_questions` (branching, FR-AW-001).

**Request**
```json
{ "answers": [ { "question_key": "system_function", "value": ["filter_applications","rank_candidates"] } ] }
```
**200** → same shape as `GET` (with refreshed `next_questions`).

### `POST /assessments/{id}/classify`
Run the deterministic engine on current answers (FR-AW-004, FR-RE-*). Locks `rule_version`/`source_version` snapshot on success.

**200 — classified**
```json
{
  "assessment_id": "aia_123",
  "classification_status": "classified",
  "risk_tier": "high_risk",
  "confidence": "high",
  "primary_actor_role": "provider",
  "secondary_actor_roles": ["deployer_when_used_internally"],
  "triggered_rules": [ { "rule_code": "annex_iii_employment_recruitment_selection",
    "source": "Regulation (EU) 2024/1689 Annex III point 4(a)", "rationale": "..." } ],
  "free_preview": { "top_obligations": ["...","..."], "document_gap_preview": ["risk_memo","annex_iv_tech_doc"] },
  "rule_version": 7,
  "source_version": "2024/1689-consolidated-2026-01"
}
```

**200 — insufficient_information**
```json
{ "assessment_id": "aia_123", "classification_status": "insufficient_information",
  "missing": [ { "question_key": "affects_natural_persons", "label": "Does it affect natural persons?" } ] }
```

**200 — needs_expert_review / conflicting_rules** include `reasons[]` and the partial trace, never a forced tier.

### `GET /assessments/{id}/result`
Returns the stored `ClassificationResult` (free preview for anonymous; full trace gated by entitlement). Without entitlement, paid sections are replaced by `{ "locked": true, "unlock_sku": "evidence_pack" }`.

## 4. Documents (PRD §14.5, §10.3)

### `POST /documents/generate`
Enqueue generation for an entitled assessment. **Requires** active `entitlement` (else `403 entitlement_required`). Idempotent per `(assessment_id, sku)`.

**Request**
```json
{ "assessment_id": "aia_123", "sku": "evidence_pack" }
```
**202**
```json
{ "report_id": "rep_789", "status": "queued" }
```

### `GET /documents/{report_id}`
Poll generation status + artifact list.

**200**
```json
{
  "report_id": "rep_789",
  "status": "ready",
  "rule_version": 7,
  "source_version": "2024/1689-consolidated-2026-01",
  "artifacts": [
    { "document_type": "risk_classification_memo", "format": "md", "download": "/downloads/sig_abc" },
    { "document_type": "evidence_pack_zip", "format": "zip", "download": "/downloads/sig_zip" }
  ],
  "warnings": [ { "type": "missing_variable", "field": "incident_response_owner" } ]
}
```
`status ∈ queued | generating | ready | failed`. On `failed`, `error` describes cause; user + admin notified (PRD §11.4).

### `GET /downloads/{signed_token}`
Streams the artifact via a short-lived signed R2 URL (TDD §12). Tokens are user- and document-scoped and expire (PRD §11.3, §21.2). `410 gone` after expiry.

## 5. Payments (PRD §14.5, §7.5)

### `POST /checkout/session`
Create a hosted Dodo Payments checkout session for a SKU. In local dev mode, with no payment provider configured, this creates an active dev entitlement and redirects back to the app.

**Request**
```json
{ "assessment_id": "aia_123", "sku": "evidence_pack", "success_url": "...", "cancel_url": "..." }
```
**200**
```json
{ "checkout_url": "https://checkout.dodopayments.com/...", "session_id": "checkout_..." }
```

### `POST /dodo/webhook`
Dodo Payments → us. Signature-verified via `webhook-id`, `webhook-timestamp`, and `webhook-signature`; idempotent by `webhook-id`. Handles `payment.succeeded` by activating the matching entitlement. Replays return `200`.

### `POST /stripe/webhook`
Legacy Stripe endpoint. Signature-verified, idempotent by event id (TDD §7.5). Handles `checkout.session.completed`. Creates/updates `entitlement`. Replays return `200`.

## 6. Admin (PRD §14.5, §10.5) — `require_role("admin")`

### Legal sources
- `POST /admin/legal-sources` — create draft source (FR-ADMIN-001).
- `PATCH /admin/legal-sources/{id}` — edit/supersede; writes `audit_log`.
- `POST /admin/legal-references` / `PATCH /admin/legal-references/{id}` — manage atomic citations.

### Rules & obligations
- `POST /admin/rules` — create rule in a draft `rule_set` (FR-ADMIN-002).
- `PATCH /admin/rules/{id}` — edit draft rule.
- `POST /admin/obligations` / `PATCH /admin/obligations/{id}`.

### Rule preview (FR-ADMIN-003)
`POST /admin/rules/preview`
```json
{ "rule_set_version": 8, "fixtures": ["resume_screening","credit_scoring","spam_filter"] }
```
**200**
```json
{ "results": [ { "fixture": "resume_screening", "expected_tier": "high_risk", "actual_tier": "high_risk", "pass": true } ],
  "all_pass": true }
```

### Change-impact analysis (FR-ADMIN-004)
`GET /admin/rules/impact?rule_set_version=8`
```json
{ "affected_seo_pages": ["hr-tech/resume-screening"], "affected_templates": ["annex-iv-technical-documentation"],
  "affected_assessments": 42 }
```

### Publish (PRD §9.4)
`POST /admin/rules/publish`
```json
{ "rule_set_version": 8 }
```
Guarded: rejects (`409 conflict`, `details: failing_fixtures`) unless the full fixture suite passes (PRD §21.1). On success: sets version `active`, supersedes previous, enqueues `regenerate_pages`, schedules customer notices.

### pSEO regeneration
`POST /admin/seo-pages/regenerate`
```json
{ "scope": "all" }   // or { "slugs": ["hr-tech/resume-screening"] }
```
**202** `{ "job_id": "job_...", "queued_pages": 312 }`.

## 7. System

- `GET /healthz` → `200 {"status":"ok"}` (liveness).
- `GET /readyz` → checks DB/Redis/R2; `503` if any dependency down.
- `GET /openapi.json` → machine spec (drives client generation).

## 9. Idempotency & concurrency rules

1. Mutating POSTs require `Idempotency-Key`; replays return the original response.
2. `classify` is naturally idempotent for identical answers + ruleset (deterministic).
3. `documents/generate` dedupes per `(assessment_id, sku)`; a second call returns the existing `report_id`.
4. Payment webhooks dedupe by provider event id.
5. `rules/publish` is transactional; concurrent publishes serialize on the single-active-ruleset constraint.
